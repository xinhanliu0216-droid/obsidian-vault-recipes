# 隐私与安全实践

本仓库只规定工作流边界，不能替你判断模型提供方是否适合处理敏感资料。处理个人、医疗、工作或客户内容前，先确认 Agent 的数据流和隐私政策。

## 导入前

- 将密钥、令牌、身份证号、账号、联系方式、精确地址和客户标识移出 `raw/`。
- 对必须保留的敏感字段使用一致的假名，而不是只遮住部分字符。
- 不确定是否允许上传的工作资料，默认不要交给云端模型。
- 把附件、导出文件和隐藏目录也纳入检查，不要只看 Markdown。

## Agent 权限

- 默认只授权当前 Vault，不授权上级目录、桌面或整个主目录。
- 默认禁止联网补充资料；需要联网时，先说明目标网站和用途。
- 批量写入、移动、重命名、覆盖和删除前，要求展示目标文件清单或 diff。
- `raw/` 保持只读；中间解析结果写入可重建的 `_staging/`。

## 共享与 Git

提交或分享前至少检查：

- `.env`、配置文件、历史导出和附件中是否含凭据；
- frontmatter、日志和绝对路径是否暴露姓名或设备信息；
- Git 历史中是否曾提交敏感内容；
- 示例数据是否真正脱敏，组合字段是否仍能识别个人；
- 公开仓库是否需要排除整个 `raw/` 或个人场景目录。

删除当前文件并不能删除 Git 历史中的秘密。若凭据曾提交，先撤销或轮换凭据，再清理历史。

## 高风险内容

- 医疗、心理和安全内容只做资料整理，不生成诊断。
- 法律、财务和人事结论必须保留来源、适用范围和人工复核状态。
- 个人模式必须同时记录支持证据、反例和不确定性，不能由一次事件推断人格。

## 发生意外时

1. 立即停止 Agent 写入和同步。
2. 保存日志与变更清单，不继续覆盖现场。
3. 从版本控制或备份恢复受影响文件。
4. 若有凭据泄露，先轮换凭据。
5. 按仓库 [安全策略](../SECURITY.md) 私下报告工具漏洞，不附真实 Vault。
