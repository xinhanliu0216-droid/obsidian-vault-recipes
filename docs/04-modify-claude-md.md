# 第 4 章：修改和演化 CLAUDE.md

`CLAUDE.md` 是可执行约定，不是愿望清单。每次修改都应该对应一个观察到的问题。

## 1. 规则应该放在哪里

| 内容 | 推荐位置 |
|---|---|
| 每次会话都适用的目标、边界、目录 | `CLAUDE.md` |
| 某类文件才适用的规则 | 嵌套 `CLAUDE.md` 或 `.claude/rules/` |
| 多步骤、按需执行的流程 | Skill |
| 个人路径、私有偏好 | `CLAUDE.local.md`，加入 `.gitignore` |
| 只有一次的要求 | 当前对话 |

目标是让根 `CLAUDE.md` 保持简短。官方建议控制在 200 行以内；拆成 import 只改善组织方式，并不会减少启动上下文。

## 2. 一次规则变更

1. 写下具体症状，例如“Source 页经常没有原文件链接”。
2. 找到负责该行为的最小规则。
3. 写成可观察指令，例如“Source 页的 `source_file` 必须是 `raw/` 下的相对路径”。
4. 增加一个能复现问题的测试案例。
5. 让 Agent 只提出 diff，不直接修改。
6. 确认后应用，并重新运行案例。

## 3. 修改前提示词

```text
请审查 CLAUDE.md，并针对下面的失败现象提出最小修改：
{失败现象}

要求：
1. 指出修改哪条规则；
2. 给出修改前后对照；
3. 说明如何测试；
4. 暂时不要修改文件。
```

## 4. 好规则的四个部分

- **触发条件**：什么时候适用
- **动作**：Agent 要做什么
- **边界**：哪些内容不能动
- **验证**：如何知道做对了

示例：

> 摄取 `raw/` 中的新文件时，创建一个 Source 页，并在 frontmatter 的 `source_file` 记录原文件相对路径。不要改写原文件。完成后验证该路径存在。

## 5. 应删除或下沉的规则

- 无法观察是否执行的抽象口号
- 与其他规则重复或冲突的内容
- 只属于某个场景却放在通用模板中的内容
- 很少触发的长流程
- 通过脚本或 Hook 才能可靠保证的操作

## 6. 版本记录

规则发生实质变化时，在 `log.md` 追加：

```markdown
## [YYYY-MM-DD] schema | 简短标题

- 原因：
- 变更：
- 测试：
- 结果：
```
