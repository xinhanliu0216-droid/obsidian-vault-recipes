# 3D：项目管理 LLM Wiki

项目 Wiki 的价值不是复制任务管理软件，而是把会议、决定、风险、交付和状态依据连接起来。当有人问“为什么这样决定”“这个任务真的完成了吗”，Wiki 能回到明确来源。

## 1. 这个场景解决什么问题

- Agent 自动给任务分配负责人；
- 时间过去就把任务标成完成；
- 讨论倾向被写成正式 Decision；
- 新周报覆盖旧状态，历史消失；
- 风险和已经发生的问题混为一谈；
- 状态报告没有数据截止时间；
- 多份会议记录冲突时自动选择最新一份。

项目场景规则必须强调：状态来自证据，不由 Agent 猜测。

## 2. 示例知识库

```text
raw/
├── project-charter.md
├── roadmap-v1.md
├── meeting-2026-07-01.md
├── meeting-2026-07-15.md
├── weekly-status-2026-W28.md
├── change-request-003.md
└── acceptance-import-performance.md
```

主要输出：Decision、Task、Risk、Milestone、状态报告和阶段复盘。

## 3. 材料怎样组织

- 项目章程定义边界、角色和成功标准；
- 每次会议一个 Source，记录日期、参与人和会议性质；
- 周报记录数据截止时间和提供人；
- 变更请求保留版本、批准状态和影响；
- 验收记录与任务状态分开；
- 长项目用 manifest 按阶段记录历史材料覆盖情况。

## 4. 页面模型和创建判断

### Decision

必须由有权角色明确确认。记录决定、背景、替代方案、依据、反对意见、生效范围和复查条件。

### Task

至少有动作、交付物、负责人、状态、来源；截止日期和依赖缺失时标待确认，不补写。

### Risk

尚未发生的不确定事件，包含触发信号、概率、影响、责任人和缓解措施。

### Issue

已经发生的问题。虽然可放 `wiki/topics/`，但不能继续标 Risk。

### Milestone

有可验证验收标准的阶段目标，不是一个模糊日期。

### Synthesis

状态报告和阶段复盘，必须说明资料截止日期和未更新来源。

## 5. 摄取会议记录

```text
请读取 meeting-2026-07-15.md 和当前项目 Wiki，先不要写文件。
区分：
- 事实更新；
- 讨论和建议；
- 明确决定；
- 行动项；
- 风险；
- 已发生问题；
- 缺少负责人、日期或确认人的事项。

列出准备更新的 Source、Decision、Task、Risk、Milestone 和 index/log。
```

如果会议说“可以考虑让小王处理”，不能直接建立 `owner: 小王`。如果说“由负责人确认下周完成”，但不知道负责人是谁，应标待确认。

## 6. 状态历史怎样维护

Task 页面不要只保留当前状态。建议保留：

```markdown
## 状态历史

- 2026-07-01：planned，来源 [[meeting-2026-07-01]]
- 2026-07-08：blocked，测试数据未到位，来源 [[weekly-status-2026-W28]]
- 2026-07-15：in-progress，数据已到位，来源 [[meeting-2026-07-15]]
```

Decision 被推翻时，旧页面保留 `superseded` 状态并链接新 Decision，不改写历史。

## 7. 长项目材料的回填

迁移已有项目时，不要一次读完所有会议记录：

1. 先按阶段和日期建立 manifest；
2. 优先摄取项目章程和关键 Decision；
3. 再处理任务、风险和周报；
4. 每批记录覆盖日期；
5. 冲突记录进入待确认清单；
6. 历史回填完成前，状态报告声明资料不完整。

## 8. 常用查询提示词

### 状态报告

```text
根据截至 2026-07-15 已摄取的材料生成状态报告。
按已完成、进行中、阻塞、风险、待决事项和下一步组织。
每项标来源日期；缺少最新状态的任务单独列出。
```

### 决定追溯

```text
为什么项目决定先建立性能基线再优化？
列出 Decision、替代方案、证据、反对意见和后续状态变化。
```

### 逾期检查

```text
列出截止日期早于今天且没有完成证据的 Task。
不要自动标记逾期或失败；区分日期明确、状态陈旧和资料尚未摄取。
```

## 9. CLAUDE.md 场景规则逐段解释

- **目标**：强调项目事实与来源，而不是代替任务系统。
- **输入单位**：区分会议、周报、变更和验收。
- **页面类型**：把 Decision、Task、Risk、Issue 和 Milestone 分开。
- **创建条件**：禁止猜测负责人、日期和状态。
- **状态规则**：要求追加历史，不覆盖过去。
- **查询方式**：强制说明数据截止时间。
- **完成标准**：检查授权、交付物和冲突记录。

完整文件：[项目 CLAUDE.md 样例](../../examples/claude-md/project-CLAUDE.md)。

## 10. 怎样改成你的项目

需要明确：

- 谁有决策权；
- Task 状态枚举；
- 哪个系统是负责人、日期和完成状态的权威来源；
- 逾期怎样定义；
- 风险等级；
- 周报或状态报告格式；
- 多项目是否共用一个 Vault。

交给 AI：

```text
请根据以下项目治理方式修改 project-CLAUDE.md：
{角色、状态、权威系统、汇报周期、风险等级和失败案例}

请只提供最小 diff，并测试：
1. 模糊行动项不能获得虚构 owner；
2. 讨论不能变成 Decision；
3. 旧状态不能被静默覆盖。
```

## 11. 失败案例与修复

- **自动分配 owner**：没有明确来源时字段为待确认。
- **时间推移等于完成**：完成必须有验收、负责人确认或权威系统记录。
- **讨论倾向等于 Decision**：要求授权角色和明确确认。
- **Risk 与 Issue 混淆**：以是否已经发生区分。
- **最新会议覆盖历史**：状态和决定追加版本历史。

## 12. 完成验收清单

- [ ] 每份会议、周报和变更有 Source；
- [ ] Task 有动作、交付物、负责人和来源；
- [ ] 缺失字段没有被猜测；
- [ ] Decision 有授权确认；
- [ ] Risk 和 Issue 正确区分；
- [ ] Milestone 有验收标准；
- [ ] 状态变化保留历史；
- [ ] 报告注明数据截止时间；
- [ ] 冲突记录进入待确认清单；
- [ ] index/log 已更新。
