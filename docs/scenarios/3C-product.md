# 3C：产品分析 LLM Wiki

产品 Wiki 不是反馈仓库，也不是自动生成路线图的机器。它要把用户原话、问题归纳、产品假设、数据、实验和正式决策分层，让团队知道每个判断来自哪里、还需要验证什么。

## 1. 这个场景解决什么问题

常见问题包括：

- 高频反馈直接等同于高价值需求；
- 用户说“加一个按钮”，Wiki 就建立 Feature；
- 竞品营销文案被当作已验证能力；
- 指标上涨被直接解释成某项改版导致；
- 会议中的想法被写成正式决定；
- 产品决定改变后，旧依据和反对意见消失。

产品场景的 `CLAUDE.md` 应维护“证据 → 问题 → 假设 → 验证 → 决策”的链路。

## 2. 示例知识库

假设团队正在改进数据导出功能：

```text
raw/
├── interview-01.md
├── interview-02.md
├── support-tickets-2026-06.csv
├── competitor-a-export.md
├── metric-export-failure-2026-06.md
├── experiment-export-v2.md
└── product-meeting-2026-07-15.md
```

主要输出：用户问题、功能假设、指标解释、方案比较、实验结论和正式 Decision。

## 3. 材料怎样组织

- 每次访谈作为 Source，保留日期、用户角色、使用场景和研究限制；
- 工单批次记录渠道、时间范围、去重方式和样本偏差；
- 指标快照记录定义、分母、时间窗口和分群；
- 竞品资料记录观察日期、版本和信息来源；
- 会议记录保留讨论、建议和谁有权确认决定。

大量反馈先用 manifest：

```markdown
| 批次 | 渠道 | 日期 | 原始条数 | 去重后 | 已处理 | 备注 |
|---|---|---|---:|---:|---:|---|
| 01 | 工单 | 6 月 | 240 | 183 | 60 | 企业用户偏多 |
| 02 | 访谈 | Q2 | 12 | 12 | 12 | 主动招募样本 |
```

## 4. 页面模型和创建判断

### User Problem

描述用户在某任务和上下文中遇到的问题。需要多条证据，或一个影响极高且明确标为个案的来源。

不应建页：单条“我要某功能”的请求、团队的解决方案猜测。

### Feature

记录功能目标、关联问题、约束、状态和验证方式。Feature 不是“有人提过”的同义词。

### Metric

必须记录定义、公式、分母、时间窗口、分群和数据来源。相同名称但口径不同的指标不能静默合并。

### Decision

只有授权角色确认后创建，包含替代方案、依据、反对意见、生效日期和复查条件。

### Synthesis

用于跨访谈、工单和指标形成阶段性判断，必须说明样本偏差、反证和未知。

## 5. 第一次摄取访谈

```text
请读取 interview-01.md 和当前 Wiki，先不要写文件。
请区分：
1. 用户原话；
2. 可观察的任务和阻碍；
3. 研究者解释；
4. 用户提出的解决方案；
5. 目前不能推广到其他用户的内容。

列出准备创建或更新的 Source、User Problem 或已有页面，并说明为什么。
```

合理结果应先创建访谈 Source。单份访谈通常只形成弱信号，不应立即创建“所有用户都需要批量导出”的确定性结论。

## 6. 从反馈批次到问题综合

处理工单集合时：

1. 记录渠道和时间；
2. 去除重复工单但保留数量；
3. 区分任务、症状、影响、解决方案请求；
4. 标记用户分群；
5. 与现有 User Problem 合并；
6. 记录高频、严重度和商业影响，但不混为一个指标。

提示词：

```text
请处理 manifest 中工单批次 01 的前 60 条。
先定义去重和编码方式；不要直接生成路线图。
完成后报告：问题主题、来源数量、用户分群、反例、无法判断项，以及会更新哪些 User Problem。
```

## 7. 从证据到 Decision

当团队讨论是否重做导出流程时：

```text
请根据当前 User Problem、Metric、Experiment 和相关 Source 比较方案 A 与 B。
使用相同维度：目标用户、解决的问题、证据、预计价值、成本、风险、未知、验证方式和停止条件。
这只是分析，不要创建 Decision。
```

会议确认后再输入：

```text
根据 product-meeting-2026-07-15.md，判断是否存在授权角色明确确认的决定。
如果存在，先给出 Decision 页面计划：决定、替代方案、依据、反对意见、生效时间和复查条件。
```

## 8. 常用查询提示词

### 用户问题

```text
哪些用户在什么场景下遇到导出问题？
按用户分群、任务、问题表现、影响、证据数量和来源列出。
不要把功能请求当成问题定义。
```

### 指标解释

```text
导出失败率最近下降是否能说明改版有效？
检查指标口径、时间、分群、同期变化和实验设计。
区分观察事实、可能解释和无法确认的因果。
```

### 机会排序

```text
比较当前三个 User Problem。
分别列证据强度、频次、严重度、战略关联、解决成本和未知。
不要生成单一总分，除非 CLAUDE.md 已定义评分规则。
```

## 9. CLAUDE.md 场景规则逐段解释

- **使用目标**：强调证据链，而不是自动生成产品建议。
- **输入单位**：让访谈、工单批次、指标和会议保持不同职责。
- **页面类型**：分开 User Problem、Feature、Metric 和 Decision。
- **创建条件**：防止一条反馈升级为普遍需求。
- **摄取规则**：强制分离用户原话、解释、假设和决定。
- **查询方式**：要求相同维度比较，并暴露未知。
- **完成标准**：检查来源、指标口径和决策授权。

完整文件：[产品 CLAUDE.md 样例](../../examples/claude-md/product-CLAUDE.md)。

## 10. 怎样改成你的产品

需要你决定：

- 目标用户和分群；
- 核心任务与指标；
- 什么证据足以创建 User Problem；
- 谁能确认 Decision；
- 实验与指标的权威来源；
- 竞品信息的更新频率；
- 是否需要机会评分以及评分规则。

交给 AI：

```text
根据以下产品背景修改 product-CLAUDE.md：
{产品阶段、用户、渠道、指标、决策角色和真实失败案例}

请给最小 diff，并设计三个测试：
1. 单条反馈不能变成普遍需求；
2. 指标相关性不能写成因果；
3. 会议想法不能自动变成 Decision。
暂时不要编辑。
```

## 11. 失败案例与修复

- **高频等于高价值**：把频次、严重度、商业影响拆成独立字段。
- **功能请求等于问题**：先还原用户任务、阻碍和结果。
- **竞品宣传等于事实**：记录观察来源和日期，能力标为待验证。
- **会议建议等于决定**：Decision 必须有授权角色和明确确认语句。
- **改版后指标上涨等于因果**：没有实验或机制证据时只记录同期变化。

## 12. 完成验收清单

- [ ] 每份访谈和反馈批次有 Source；
- [ ] 用户原话、解释和产品假设分开；
- [ ] User Problem 有证据和适用用户；
- [ ] Metric 有定义、口径、时间和来源；
- [ ] Feature 关联真实问题与验证方式；
- [ ] Decision 有授权来源、替代方案和复查条件；
- [ ] 样本偏差、反例和未知明确；
- [ ] 没有把相关性写成因果；
- [ ] index/log 已同步。
